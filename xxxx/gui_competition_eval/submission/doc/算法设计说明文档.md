# 算法设计说明文档

## 1. 提交包定位
本提交包基于主办方提供的 `agent_base.py` 原版基类实现，`agent.py` 仅通过 `BaseAgent._call_api()` 调用多模态模型接口，不自行创建客户端，不覆盖官方固定配置。

## 2. 官方 API 接入方式
主办方基类中已经固定了默认调试配置：
- API URL: `https://ark.cn-beijing.volces.com/api/v3`
- Model ID: `doubao-seed-1-6-vision-250815`

因此本提交包接入官方 API 的方式是：
1. 保留官方 `agent_base.py` 原样不改。
2. `agent.py` 通过 `_call_api(messages, temperature=0)` 发起调用。
3. 本地调试时仅需设置环境变量 `VLM_API_KEY`。
4. 提交阶段由主办方注入 `EVAL_*` 环境变量，代码无需改动。

## 3. Agent 框架
本实现采用“首步规则 + 单步视觉决策 + 鲁棒解析 + 轻量状态管理”的结构：

### 3.1 首步规则
如果用户指令中可显式识别目标 App，且当前为第 1 步，则直接输出 `OPEN`，降低模型首步发散概率。

### 3.2 单步视觉决策
后续步骤把用户指令、当前步数、最近动作摘要和当前截图组成消息，调用视觉模型判断当前最合理的下一步动作。

### 3.3 鲁棒动作解析器
解析器按三层顺序抽取结果：
1. JSON 风格输出
2. 显式动作语法（如 `CLICK:[[100,200]]`）
3. 逐行兜底提取

统一转换到标准格式：
- CLICK -> `{"point": [x, y]}`
- TYPE -> `{"text": "..."}`
- SCROLL -> `{"start_point": [...], "end_point": [...]}`
- OPEN -> `{"app_name": "..."}`
- COMPLETE -> `{}`

### 3.4 多步状态管理
状态管理器仅保存：
- 原始任务指令
- 目标 App 候选
- 最近 4 步动作摘要
- 解析失败计数

并在输出前做最后一层修正：
- 坐标归一化并截断到 0~1000
- 重复 CLICK / SCROLL 轻微扰动
- OPEN 无 app_name 时回填目标 App

## 4. 为什么这样设计
本赛题是逐步 GUI 操作任务，关键并不是复杂 long-CoT，而是：
- 输出格式必须稳定
- 首步打开 App 不能乱
- 多步过程中不能频繁重复动作
- 必须和官方评测协议完全对齐

因此本实现优先保证协议稳健性和评测兼容性，而不是堆叠复杂规划器。

## 5. 本地使用说明
### 5.1 安装依赖
```bash
pip install -r src/requirements.txt
```

### 5.2 设置 API Key
```bash
export VLM_API_KEY=你的key
```

### 5.3 目录结构

```text
    ├── requirements.txt

```
